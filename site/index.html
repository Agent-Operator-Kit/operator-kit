V2 keeps the model simple.

Map the project, choose the right lanes, give each lane approved patterns, and let the operator approve parallel batches before anything is dispatched.

Recommended lanes

Create durable lanes only where they help: ownership, contracts, risk, validation, or dense context. Use temporary role overlays for smaller specialist work.

Approved patterns

Role templates can carry preferred packages, repos, validation recipes, and consistency rules so agents reuse known solutions.

Approved batches

The planner proposes parallel work from dependencies and lane ownership. The operator still decides what gets sent.

many safe lanes.

Worktrees

tmux Lanes

03 — HANDOFFS

External Task State

Task packets, handoffs, screenshots, and scratch files stay under OPERATOR_DIR, outside the repo.

04 — EVERGREEN

Clean Codebase

The repo keeps source, reusable scripts, stable rules, and durable docs.

05 — MOMENTUM

Feature Tracks

The operator keeps routing follow-ups until the work is integrated or truly blocked.

06 — MEMORY

Relevant Context Packs

Agents get only the project, task, and episode memory relevant to their lane.

07 — UPGRADE

One Machine Refresh

Feedback To Execution

FB-* and RM-* IDs keep feedback and roadmap work traceable without bloating the repo.

- inspect the repo first
- detect whether Operator Kit is installed, partial, or missing

Use the agent
that fits the work.

Codex can operate from Desktop, Claude Code is excellent for scoped implementation lanes, and Cursor can be the cockpit or a worker. The lane model stays the same.

01 — COCKPIT

Codex · Claude Code · Cursor

Pick one cockpit for integration decisions, then assign worker lanes by strength.

02 — RULES

Shared rules

Project assets keep lane boundaries and handoff rules available to each agent surface.

03 — SKILLS

Explicit modes

Use operator, feedback, planner, and design modes instead of mixing observation, planning, and implementation.

04 — SUBAGENTS

Specialists

Use temporary subagents for research and durable lanes for branch-owning implementation.

05 — LOCAL LANES

Local workers

Run CLI workers in tmux lanes when you need persistent local execution and collectable handoffs.

06 — CLOUD AGENTS

Remote workers

Remote agents need the full task packet because they cannot read local OPERATOR_DIR.

// feedback planning execution

Separate observation
from implementation.

The explicit mode split keeps testing lightweight. Annotations are feedback by default; feedback becomes execution only after planner promotion, V2 dependency metadata, lane ownership, and operator approval.

01 — FEEDBACK MODE

$operator-feedback

Captures testing notes, browser annotations, mobile screenshots, videos, coordinates, testIDs, severity, type, and expected behavior as structured FB-* intake.

02 — PLANNING MODE

$operator-planner

Reviews feedback, groups duplicates, prioritizes now/next/later, promotes selected work into RM-*, and adds required roles, owner lanes, contracts, dependencies, and approval gates.

03 — EXECUTION MODE

$operator

Runs the batch planner, creates task folders, checks lane and file ownership, dispatches approved scoped packets, collects handoffs, reviews diffs, and integrates implementation work.

04 — MOBILE LOOP

Simulator Review

Capture iOS Simulator screenshots or recordings into a feedback task, open the local review UI, click screenshot locations, annotate, then triage into the feedback inbox.

05 — TRACE NOTES

Link Code Back To Why

Keep roadmap files local, then add a short PR or commit note: Roadmap, Feedback, Operator task, Why, and Validation.

06 — MULTI-DEV

Shared Planning Surface

When collaboration starts, move RM-* and accepted FB-* items to a planning repo or tracker. Each operator keeps execution files local, while PRs carry trace IDs.

mode examples Roadmap concept
# feedback is observation, not execution
Use $operator-feedback with $design-agent. Capture these annotations.

// upgrade command

Refresh every agent surface
with one command.

operator-upgrade.sh discovers installed Operator Kit projects, updates project scripts/templates, installs V2 catalog and system-map support, and runs checks. Codex Desktop skills are refreshed when available; Cursor-only machines can use --skip-skills.

01 — SKILLS

Optional Codex Desktop

Refreshes $operator-feedback, $operator-planner, $operator, $design-agent, $incubation, and setup guidance in ~/.codex/skills when Codex is part of your stack.

02 — DISCOVER

Installed projects

Scans ~/Projects by default for operator.config.env. Use --target or --projects-root to scope it.

03 — PRESERVE

Project state

Preserves config, source code, handoffs, task packets, task working files, memory, captures, and existing project docs.

04 — VERIFY

Checks after update

Runs syntax, lane status, summary, memory, roadmap, catalog, lane recommendation, batch-plan, and git status checks for each updated project.

upgrade examples Open script
# preview changes
bash scripts/operator-upgrade.sh --dry-run
 
# refresh this machine
bash scripts/operator-upgrade.sh

Use $operator /upgrade.
