// model

One operator,
many isolated lanes.

A lightweight operating model that keeps your codebase clean while agents run in parallel across named worktrees.

01 — ISOLATION

Worktrees

Each agent lane gets a separate directory and branch. Long-running work stays isolated and never conflicts with other lanes or main.

02 — PERSISTENCE

tmux Lanes

Persistent terminals keep Codex, Claude Code, and Cursor CLI workers visible, inspectable, and recoverable across sessions.

03 — HANDOFFS

External Task Packets

Task packets, pane captures, and raw handoffs live under an operator workspace outside the repo — delete or archive without touching git history.

04 — EVERGREEN

Clean Codebase

The repo stores only source code, reusable scripts, stable rules, and durable docs. Generated operator state never bleeds into source history.

05 — MOMENTUM

Feature Tracks

Once a feature is authorized, the operator keeps routing lane follow-ups until it is complete, validated, integrated, or blocked by a real user decision.

Open Codex, Claude Code, or Cursor in the target repo and paste the prompt on the right. The agent handles the rest.

Open your agent
Launch Codex, Claude Code, or Cursor in the target repository.

Review and commit

$ cd operator-kit
$ bash scripts/operator-bootstrap.sh /path/to/your/repo

- create the external operator workspace outside the repo

// codex desktop

Use $operator
as the project cockpit.

Install the runtime skill globally, then ask Codex to operate installed Operator Kit projects in natural language. The skill detects the kit first and uses the project-local scripts as source of truth.

01 — INSTALL

Global Codex Skill

Copy skills/codex/operator/SKILL.md to ~/.codex/skills/operator/SKILL.md, then reopen Codex Desktop so $operator is available.

02 — DETECT

Installed, Partial, Missing

The skill checks upward and across sibling worktrees for operator.config.env plus the required scripts/operator-*.sh commands before it acts.

03 — OPERATE

Status, Dispatch, Collect, Update

For installed projects it reads config, runs status and summary, creates task packets under OPERATOR_DIR, dispatches lanes, reviews handoffs, and refreshes kit assets safely.

04 — GUARD

No Unsafe Guessing

For partial or missing installs it reports what is missing and avoids tmux dispatch or collection until the Operator Kit install is repaired.

// design-agent companion

Add $design-agent
for UX and design-system work.

Use the optional Codex Desktop design-agent skill to review product UX, extract or recommend a design system, classify web-preview annotations, and shape Claude Code Opus or Operator Kit UI tasks.

01 — REVIEW

UX and consistency audits

Ask for a comprehensive review when you want drift, awkward flows, weak states, copy issues, and component inconsistencies called out before implementation work starts.

02 — SYSTEM

Design-system extraction

For existing products, it inspects code, tokens, components, CSS variables, and screens, then separates intentional design language from accidental drift.

03 — STARTER

Starter recommendation

For new projects, it recommends a grounded starter such as shadcn/Radix/Tailwind, Material UI, Ant Design, Mantine, or Chakra instead of letting every agent improvise.

04 — OPERATOR

Task shaping, not dispatch

$design-agent owns design judgment and task content. $operator owns project detection, lane safety, dispatch, collection, and integration review.
