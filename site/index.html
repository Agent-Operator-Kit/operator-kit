Value

Use it when one agent is no longer enough.

Real projects have more than frontend and backend. They have provider integrations, mobile apps, auth, storage, deployment, knowledge bases, observability, and release work. Operator Kit gives each slice a clear boundary so agents can specialize without trampling each other.

01

More parallel work

Run multiple agents at once by assigning each one a lane, branch, task packet, and validation target.

02

Cleaner source control

Keep raw handoffs, screenshots, scratch notes, memory, and task files outside the app repo.

03

Better consistency

Use approved patterns, role templates, and lane memory so agents reuse the same project decisions.

Approach

The hard part is cutting the system into useful lanes.

More lanes are not automatically better. A lane is worth creating when it gives an agent sharper context, durable memory, clear ownership, and a different validation loop. Smaller specialist roles can stay as temporary overlays until they deserve their own branch and worktree.

Long-lived ownership The work will keep coming back and should accumulate memory.
Explicit contract boundary The lane owns APIs, data contracts, provider contracts, or product surfaces.
High-risk domain Auth, money movement, release, privacy, production recovery, or provider sync.
Distinct validation loop Mobile simulator, backend tests, LLM evals, provider sandbox, or deployment checks.
High context density The lane benefits from focused docs, memory, approved patterns, and local examples.

Each lane template can carry an approved toolkit.

  • Recommended frameworks, packages, GitHub repos, and examples.
  • Approved architecture patterns such as retries, caching, RLS, logging, or API contract style.
  • Validation commands and handoff requirements for the lane.
  • Escalation gates for risky changes, credentials, providers, deploys, and data migrations.
  • Project-specific memory so agents do not rediscover the same decisions every session.
How it works

A simple loop for controlled parallel work.

The operator is the integration point. Worker agents do scoped work in isolated lanes. The operator reviews handoffs, promotes durable memory, checks dependencies, and merges only approved source changes.

1. Map Identify the system areas, contracts, agents, and validation commands.
2. Recommend Choose durable lanes and attach approved patterns, tools, and validation rules.
3. Approve Plan a dependency-safe batch instead of dispatching the roadmap linearly.
4. Integrate Collect handoffs, inspect diffs, capture memory, and merge accepted source changes.

Start

Install it by asking an agent.

Open Codex, Claude Code, Cursor, or another coding agent in the target repo and paste the prompt. The agent should inspect first, install or migrate the kit, recommend lanes, and report what changed.

Manual command

From a local checkout

Clone this repository, then run bash scripts/operator-sync.sh --target /path/to/repo --bootstrap-if-missing.
