Operator Kit / coding agents

The Operator model for specialized agents.

Why add it

Agents write code. Operator Kit coordinates the work.

If you already live in Codex, Claude Code, and Cursor, the hard part is no longer getting code generated. The hard part is deciding which agent owns which slice, what context it should remember, which patterns it must follow, which tasks can run in parallel, and what is safe to merge.

Lane ownership

One agent, one boundary.

Each lane has its own branch, task packet, owned files, validation target, and handoff expectations.

Shared memory

Project decisions survive the chat.

Durable facts, handoff summaries, and lane context live outside the app repo but stay available to agents.

Approved patterns

Agents reuse the architecture.

Role templates can carry recommended packages, repos, frameworks, validation commands, and solution patterns.

Batch control

Parallel work stays reviewable.

The operator approves non-conflicting work groups instead of launching autonomous agents into the same files.

Lane strategy

Split by context, contract, risk, and validation.

More lanes are not automatically better. Create a durable lane only when the agent needs a different context pack, ownership boundary, risk model, or proof of correctness. Smaller specialists can stay as role overlays until the work repeats enough to deserve its own branch and memory.

01

Contract

The lane owns a surface such as API, data, auth, mobile, provider sync, or release.

02

Context

The work has dense local knowledge, docs, decisions, and approved patterns worth remembering.

03

Risk

The lane touches credentials, payments, privacy, deploys, production recovery, or external providers.

04

Validation

The work needs a different proof loop: simulator, backend tests, provider sandbox, evals, or smoke checks.

Lane templates act like architecture patterns for agents.

  • Recommended frameworks, packages, GitHub repos, and reference implementations.
  • Approved solution patterns such as retries, caching, RLS, logging, API contracts, and release gates.
  • Validation commands, handoff requirements, escalation rules, and project-specific memory.
